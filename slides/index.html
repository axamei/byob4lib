Modularity, FTW

  • FTW = For the Web. For the Win.
  • break out into individual modules/pieces
  • for long term maintainability and sanity. Keep it organized! keep it DRY. Avoid duplication
  • Consistency. Reuse. Time saving.
  • Write code that you can come back to and understand months later - often an afterthought particularly with client side/css code - use CSS stats as a possible example
  • Isolation. Debugging. Don’t waste hours of your life trying to find the piece of code you want to override.
  • Lighten your markup/css footprint. Refactor
  • Also, let’s you grab what you want, grab what you need, or grab the lot; the choice is yours.
  • Feel like you are coding again. It's more engaging and fun!
  • Examples of modular apps libraries use: Project Blacklight. Drupal 8

  • What our current base environment looks like: [Level One]Git, Node and npm, Ruby, [Level Two] Bower / Bundler, Grunt / Gulp[are we using this one]?, Livereload, [Level three] Compass, Sass Globbing, other Sass Utitlies
  • Modularity = lots of parts. So we use task runners and package managers to have a manageable workflow.
  • Task automation / build tool / do things faster: Vendor prefixing, Sprite generation, Linting, Image Optimization, Livereload, Concatenation, Minification, Compile Sass, etc. etc. SO MANY PLUGINS!
  • Use something that works for you.. any of these tools can help you do that
  • Bower and Bundler: Project dependency management aka package management
  • Given the churn of front-end dev tools in six months we might be using libsass and Broccoli.
  • Under this model you can swap in new better tools whenever you want. Don't like how the grid is managed, get a new one.
  • These tools are helping us take a more granular approach to modularity than a highly prescriptive framework like Bootstrap requires.
  • One that gives us the "Killer" Bootstrap Features and alternative solutions / tools to allow creative freedom and rapid iteration
  • You fully control your markup and classes (or can tailor your styles to match those you may have)

  • Abstractions contains Sass mixins, extends and functions for use throughout your stylesheets and help to promote code reuse.
  • Base styles define the default look for HTML and, to some extent, common elements. These rules will generally be made up of element selectors for HTML elements such as headings, paragraphs and lists.
  • Components: This directory should contain full components (modules in SMACSS), their sub-components and modifiers. Components are discrete parts of your page that should sit within the regions of your layouts. You should try to abstract your components as much as possible to promote reuse throughout the theme.

Preventing Deviations

SCSS:


$font-stack:    Helvetica, sans-serif;
$primary-color: #333;

body {
  font: 100% $font-stack;
  color: $primary-color;
}

CSS:


body {
  font: 100% Helvetica, sans-serif;
  color: #333;
}
  • Variables for maintainability. Fewer deviations. Color/Font, etc deviations. The same thing slightly different over and over again.
  • This feature can add a lot of value to your code, in terms of maintainability, and in cases where you are required to make global changes in a single go. Here is how you declare a variable using SASS: “$primary-color:#333;” Here ‘primary-color’ is the variable name and #333, is the color value that is stored in it.
  • You can assign numbers, strings of text, colors, boolean and list of values separated by commas, to a variable.
  • no more hunt and peck
  • Minimizing deviations also promotes lower specificty. Keep your selectors short. Longer selectors have a higher specificity. Specificity is a nightmare and you should keep specificity as low as possible all of the time. We already know that we do not use IDs in CSS but a chain of selectors are often just as bad (though not quite).
  • Reducing selector length reduces selector specificity and that is very important. High specificity leads to self-induced specificity battles that can only be won by making subsequent selectors more specific (or using !important, shame on you).
  • No one wins in an !important war

Selector Inheritance

SCSS:


nav {
  ul {
    margin: 0;
    padding: 0;
    list-style: none;
  }

  li { display: inline-block; }

  a {
    display: block;
    padding: 6px 12px;
    text-decoration: none;
  }
}

CSS:


nav ul {
  margin: 0;
  padding: 0;
  list-style: none;
}

nav li {
  display: inline-block;
}

nav a {
  display: block;
  padding: 6px 12px;
  text-decoration: none;
}
  • Nesting / selector inheritance
  • Grouping selectors. Child inherits what the parents have.
  • Can get rather deep nesting. Try to avoid increasing specificity
  • Warning: inception rule / don’t go more than 4 levels deep coz it gets messy. and too specific.

Reusable Code

SCSS:


@mixin notification {
	padding:10px;
	border-radius:5px;
	font-size:1em; 
}

.error {
	@include notification;
	background:red;
	color:white; 
}

CSS:


.error {
	padding:10px;
	border-radius:5px;
	font-size:1em;
	background:red;
	color:white; 
}
  • Mixins improve reusability
  • One of the core attributes of a good CSS code is reusability.
  • Find and chunk out resuable.
  • Mixins are reusable sets of styles which can be used throughout the stylesheet. Correct usage of mixins can reduce the usage of non-semantic stray classes like ‘border-red’, ‘no-padding’ etc.
  • Here, the first section of code, declares a mixin named ‘notification’, which has a set of 3 rules defined inside. In the second section of code, the selector ‘.error’ reuses the mixin, along-with its own rules. ‘@include’ includes the mixin rules within a selector’s rule sets
